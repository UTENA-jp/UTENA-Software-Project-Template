# Development Rules

このファイルは、開発時に守るルールをまとめるテンプレートです。

## UTENA共通ルール

- 会話履歴に依存した書き方をしない
- 初めて読む人でも理解できるように書く
- 現在の状態を必ず書く
- 次にやることを必ず書く
- 未確認のことを「完了」と書かない
- 新機能を追加する前に、不具合や未確認事項を減らす
- 仕様変更・実装変更をしたら、ソースコードとドキュメントを同時に更新する
- ChatGPTやCodexへ渡す前提で、ドキュメント単体で意味が通じるように書く

## 役割分担

| 役割 | 担当 | 内容 |
| --- | --- | --- |
| プロダクトオーナー | タカシ | 目的、優先順位、最終判断 |
| 設計相談 | ChatGPT | 仕様整理、設計案、リスク整理 |
| 実装 | Codex | コード修正、動作確認、GitHub反映 |

## ドキュメント更新ルール

コードを変更したら、必要に応じて次のファイルも更新します。

- `docs/PROJECT_OVERVIEW.md`
- `docs/CURRENT_STATUS.md`
- `docs/ROADMAP.md`
- `docs/KNOWN_ISSUES.md`
- `docs/CHANGELOG.md`
- `docs/HANDOFF.md`

特に必ず更新するもの:

- 現在の状態が変わったら `CURRENT_STATUS.md`
- 次にやることが変わったら `ROADMAP.md`
- 不具合を見つけたら `KNOWN_ISSUES.md`
- 作業を完了したら `CHANGELOG.md`
- 引き継ぎが必要なら `HANDOFF.md`

## 禁止事項

- 「前に話した通り」とだけ書く
- 会話を読まないと分からない省略をする
- 未確認のことを「完了」「できた」と書く
- 動作確認していないのに「確認済み」と書く
- 仕様書と実装をずらしたままにする
- 不具合を隠して新機能を追加する

## 実装前チェック

作業前に確認すること:

1. `README.md` を読む
2. `docs/PROJECT_OVERVIEW.md` を読む
3. `docs/CURRENT_STATUS.md` を読む
4. `docs/KNOWN_ISSUES.md` を読む
5. `docs/ROADMAP.md` を読む

## 実装後チェック

作業後に確認すること:

- アプリが開くか
- 変更した機能が実際に動くか
- 既存の主要機能が壊れていないか
- エラーが出ていないか
- ドキュメントを更新したか
- Gitの状態を確認したか

## コミットルール

記入例:

```bash
git status
git add .
git commit -m "Fix task save issue"
git push
```

コミットメッセージは、何をしたか分かる短い英語または日本語にします。

## ChatGPT / Codex への依頼テンプレート

```text
README.md と docs/ 以下を読んでから作業してください。

目的:
〇〇を修正したい。

条件:
- 会話履歴に依存しないで判断してください。
- 未確認のことを完了扱いしないでください。
- 新機能より不具合修正を優先してください。
- 修正後はソースコードとドキュメントを両方更新してください。
- 最後に現在の状態と次にやることを docs に反映してください。
```
